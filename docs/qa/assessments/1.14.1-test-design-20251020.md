# Test Design: Story 1.14.1 - Fix Document Upload & N8N Webhook Integration

**Date:** 2025-10-20
**Designer:** Quinn (Test Architect)
**Story Status:** Draft
**Priority:** Critical

---

## Executive Summary

This test design covers the critical document processing pipeline fix that enables the core RAG functionality of PolicyAi. Without this working, uploaded documents have no extracted text, no vector embeddings, and chat cannot use RAG - making the system non-functional for its primary use case.

### Test Strategy Overview

- **Total test scenarios:** 34
- **Unit tests:** 8 (24%)
- **Integration tests:** 18 (53%)
- **E2E tests:** 8 (23%)
- **Priority distribution:**
  - P0 (Critical): 18 scenarios
  - P1 (High): 12 scenarios
  - P2 (Medium): 4 scenarios
  - P3 (Low): 0 scenarios

### Risk Context

**System Impact:** This bug fix is mission-critical. The document processing pipeline is currently completely broken:
- Documents upload but never get processed
- No text extraction occurs
- No vector embeddings are generated
- Chat cannot use RAG (Retrieval-Augmented Generation)
- Users cannot get AI responses based on their documents

**Test Focus:** Heavy emphasis on integration tests (53%) due to multi-service orchestration (Frontend → Edge Function → N8N → Callback → Database → Real-time UI).

## Test Execution Strategy

### Phase 1: Pre-Deployment Validation (CI Pipeline)

**Execute before deployment:**

1. **All P0 Unit Tests** (5 tests, ~5 seconds)
   - Fast feedback on critical logic
   - Must pass before any deployment

2. **All P0 Integration Tests** (15 tests, ~2 minutes)
   - Validate component interactions
   - Use mocked N8N endpoints

3. **Critical E2E Tests** (3 tests: 1.14.1-E2E-001, E2E-003, E2E-007, ~5 minutes)
   - Requires staging environment with test N8N instance
   - Validates critical paths only

**Total CI Time:** ~8 minutes for P0 tests

### Phase 2: Post-Deployment Validation (Staging)

**Execute after deployment to staging:**

1. **All P1 Tests** (12 tests, ~10 minutes)
   - More comprehensive coverage
   - Real environment validation

2. **Remaining E2E Tests** (5 tests, ~8 minutes)
   - Full user journey validation
   - Visual regression tests

**Total Staging Time:** ~18 minutes

### Phase 3: Production Smoke Tests

**Execute after production deployment:**

1. **Smoke Test Suite** (4 tests, ~3 minutes)
   - Upload single document
   - Verify processing completes
   - Verify chat works with document
   - Verify no port redirect

## Test Maintenance Considerations

### High Maintenance Risk Areas

1. **E2E Tests with Real N8N**
   - **Risk:** N8N workflow changes break tests
   - **Mitigation:** Use dedicated test workflows, version control N8N configs

2. **Real-time Subscription Tests**
   - **Risk:** Timing issues cause flaky tests
   - **Mitigation:** Use generous timeouts, retry logic in test framework

3. **Mock N8N Server**
   - **Risk:** Drift from real N8N behavior
   - **Mitigation:** Update mocks when N8N contract changes, validate against real N8N periodically

### Low Maintenance Areas

- Unit tests (stable, fast, no external dependencies)
- Integration tests with mocks (controlled environment)

## Recommendations for Dev Team

### Critical Pre-Implementation Actions

1. **Set up Mock N8N Server** before writing integration tests
   - Save hours of test development time
   - Faster test execution (no real N8N roundtrip)

2. **Configure Test Environment Variables** in CI/CD
   - Required for integration tests to run
   - Document in README

3. **Create Test PDF Documents** with known content
   - Enables content verification in RAG tests
   - Store in `tests/fixtures/documents/`

### Test-First Development Approach

For this critical bug fix, consider **test-driven development**:

1. Write failing integration test for edge function call (1.14.1-INT-001)
2. Un-comment edge function code to make test pass
3. Write failing test for N8N webhook (1.14.1-INT-003)
4. Configure webhook URL to make test pass
5. Continue pattern...

**Why TDD here?**
- Async pipeline is hard to debug manually
- Tests provide instant feedback
- Catches regressions immediately

### Testing Quick Wins

**Fastest path to confidence:**

1. Implement 1.14.1-E2E-001 (upload triggers edge function) - **5 minutes**
2. Implement 1.14.1-E2E-003 (full pipeline) - **15 minutes**
3. Implement 1.14.1-E2E-007 (RAG works) - **10 minutes**

**Total:** 30 minutes for 80% confidence

These 3 E2E tests cover the critical path. If they pass, pipeline works.

## Appendix: Test Scenario Details

### Example: 1.14.1-E2E-007 (Critical RAG Test)

```typescript
describe('1.14.1-E2E-007: Complete RAG Workflow', () => {
  it('should enable chat to use uploaded document content', async () => {
    // Given: A test PDF with known content
    const testPDF = 'tests/fixtures/vacation-policy.pdf';
    const knownContent = 'Employees are entitled to 15 days vacation';

    // When: User uploads document
    await page.goto('http://localhost:8082/dashboard');
    await page.locator('input[type="file"]').setInputFiles(testPDF);
    await page.locator('button:has-text("Upload")').click();

    // Then: Wait for processing to complete
    await expect(page.locator('.document-card')).toContainText('Completed', { timeout: 60000 });

    // When: User creates chat with document
    await page.locator('.document-card').click();
    await page.locator('button:has-text("Chat")').click();

    // And: User asks question about document content
    await page.locator('textarea[placeholder*="Ask"]').fill('How many vacation days do employees get?');
    await page.locator('button[aria-label="Send"]').click();

    // Then: AI response uses document content
    const response = await page.locator('.ai-message').last().textContent();
    expect(response).toContain('15 days'); // Content from PDF

    // And: Response includes citation to document
    expect(page.locator('.citation-badge')).toBeVisible();
  });
});
```

**Why this test is P0:**
- Ultimate validation that entire pipeline works
- Tests user-facing value proposition
- Revenue-critical functionality
- High complexity (multiple services)
