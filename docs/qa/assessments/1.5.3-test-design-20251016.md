# Test Design: Story 1.5.3 – Role Assignment UI for Company Operators

Date: 2025-10-16
Designer: Quinn (Test Architect)

## Test Strategy Overview

- Total test scenarios: 15
- Unit tests: 5 (33%)
- Integration tests: 5 (33%)
- E2E tests: 5 (33%)
- Priority distribution: P0: 6, P1: 7, P2: 2

References
- Story: `docs/stories/1.5.3.role-assignment-ui-company-operators.md`
- Frontend Spec: `docs/design/front-end-specification.md` (Admin UI, Dialogs, Tables)
- Sprint Change Proposal: `docs/project-management/sprint-change-proposal.md` (User management API, endpoints, role rules)
- AG-UI + CopilotKit Integration Summary: `docs/project-management/ag-ui-copilotkit-integration-summary.md` (General UI patterns; not directly under test for this story)

Assumptions
- API endpoints per proposal exist: `GET /api/admin/users`, `POST /api/admin/users/:id/role`, `POST /api/admin/users/bulk-role`.
- Roles: board, administrator, executive, company_operator, system_owner.
- Company Operator cannot assign `system_owner`. System Owner can assign any role including `system_owner`.
- Audit logs are persisted (table or function) and accessible via API or observable side-effect.

## Test Scenarios by Acceptance Criteria

### AC1: Dashboard lists all company users with current roles

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-UNIT-001 | unit | P1 | Transform API users → table rows shows role badges | Pure mapping/formatting logic |
| 1.5.3-INT-001 | integration | P1 | `GET /api/admin/users` returns users filtered to company scope; roles present | Contract with backend and auth/RLS |
| 1.5.3-E2E-001 | e2e | P1 | Dashboard renders sortable list; role badges visible; sorting/filters work | User flow validation |

### AC2: Company Operator can assign Board/Admin/Executive/Company Operator (NOT System Owner)

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-UNIT-002 | unit | P0 | Permission guard denies `system_owner` when current user is `company_operator` | Critical security rule |
| 1.5.3-INT-002 | integration | P0 | `POST /users/:id/role` with target `system_owner` returns 403 for operator | Enforces server-side authorization |
| 1.5.3-E2E-002 | e2e | P1 | Operator UI disables/hides `system_owner` in dropdown; allowed roles succeed | Prevents accidental elevation |

### AC3: Confirmation dialog shows role permissions before assignment

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-UNIT-003 | unit | P2 | Role → permission description mapping renders expected text | Deterministic content mapping |
| 1.5.3-E2E-003 | e2e | P1 | Dialog displays correct description and requires explicit confirm; success/error toasts | UX correctness and guardrail |

### AC4: Bulk role assignment for multiple users

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-UNIT-004 | unit | P1 | Multi-select state manager composes correct payload | Local logic correctness |
| 1.5.3-INT-003 | integration | P0 | `POST /api/admin/users/bulk-role` updates allowed roles; rejects forbidden ones | Backend bulk safety |
| 1.5.3-E2E-004 | e2e | P1 | Select N users → assign role → progress indicator → success state reflects updates | End-user workflow |

### AC5: Audit log records all role changes with timestamp and operator info

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-UNIT-005 | unit | P0 | Audit payload builder includes actor, subject, from→to role, timestamp | Data integrity for audits |
| 1.5.3-INT-004 | integration | P0 | After `POST .../role`, an audit record is persisted and retrievable (or emitted) | Compliance-critical |

### AC6: System Owners can assign any role including other System Owners

| ID | Level | Priority | Test | Justification |
|---|---|---|---|---|
| 1.5.3-INT-005 | integration | P0 | As `system_owner`, `POST /users/:id/role` to `system_owner` succeeds 200 | Confirms highest-privilege path |
| 1.5.3-E2E-005 | e2e | P1 | Owner UI shows all roles including `system_owner`; assignment flow succeeds | Highest-privilege UX + guardrails |

## Risk Coverage

- Security/Authorization (P0): 6 scenarios (UNIT-002, INT-002, INT-003, UNIT-005, INT-004, INT-005) – focuses on privilege boundaries and auditing
- Usability/Correctness (P1): Sorting/filtering, dialog content, progress indicators
- Data Integrity (P0): Audit correctness and persistence

## Recommended Execution Order

1. P0 Unit: 1.5.3-UNIT-002, 1.5.3-UNIT-005
2. P0 Integration: 1.5.3-INT-002, 1.5.3-INT-003, 1.5.3-INT-004, 1.5.3-INT-005
3. P1 E2E: 1.5.3-E2E-002, 1.5.3-E2E-005
4. P1/P2 remainder

## Notes on Tooling and Patterns

- Frontend: React Testing Library + Vitest for unit; Playwright/Cypress for E2E (project uses Vitest; choose Playwright if available)
- Integration: Exercise Supabase Edge Functions/API with test auth contexts for `company_operator` and `system_owner`; verify RLS/authorization paths per proposal
- UI Patterns: Follow dialog, table, bulk action patterns in `front-end-specification.md` (Admin dashboards, confirmation flows)

## Coverage Summary

- Every AC covered by at least one scenario
- P0 coverage concentrates on authorization boundaries and audit logging
- No duplicate tests across levels; unit validates logic, integration validates contracts/RLS, E2E validates workflows
