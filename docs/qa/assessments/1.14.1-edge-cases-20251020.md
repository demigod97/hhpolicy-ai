## Purpose

This document extends the primary test design with **23 additional edge case scenarios** that address:
- Race conditions in async pipeline
- Data corruption and recovery
- Network reliability issues
- Security boundary testing
- Scale and performance limits
- Production failure modes

These scenarios complement the existing 34 tests by focusing on **uncommon but realistic production scenarios** that could cause silent failures or data integrity issues.

## Edge Case Categories

### Category 1: Race Conditions & Timing Issues

#### EC-001: Rapid Upload-Delete Race Condition

| ID                | Level       | Priority | Test Scenario | Risk |
|-------------------|-------------|----------|---------------|------|
| 1.14.1-INT-031    | Integration | P0       | Upload document → Immediately delete before processing starts → Verify edge function handles gracefully | Critical: Could cause orphaned processing jobs |

**Scenario Details:**
```typescript
// Given: User uploads document
const upload = await uploadDocument('test.pdf');

// When: User deletes document before edge function executes
await deleteDocument(upload.id); // Race condition

// Then: Edge function should detect deleted document and exit gracefully
// And: Status should remain 'deleted', not 'processing'
// And: No orphaned N8N jobs created
```

**Why P0:** Race conditions can cause resource leaks and orphaned processing jobs that waste N8N capacity.

---

#### EC-002: Duplicate Edge Function Invocation

| ID                | Level       | Priority | Test Scenario | Risk |
|-------------------|-------------|----------|---------------|------|
| 1.14.1-INT-032    | Integration | P0       | Trigger edge function twice for same document → Verify idempotent behavior | Critical: Could cause duplicate embeddings |

**Scenario Details:**
- **Root Cause:** Database triggers, retries, or manual invocation could cause duplicate calls
- **Expected Behavior:** Second call should be no-op (idempotent)
- **Validation:** Check document has single set of embeddings, not duplicates

**Implementation Note:** Edge function should check if document already has `processing_status='processing'` or `'completed'` and exit early.

**Scenario Details:**

```typescript
// Then: Callback should handle race gracefully
// And: Final status should be 'completed' (not stuck in 'processing')
```

**Why P1:** Unlikely but possible with small documents. Could leave status stuck.

### Category 2: Data Corruption & Partial Failures

#### EC-005: Callback with Malformed Embeddings

| ID                | Level       | Priority | Test Scenario | Risk |
|-------------------|-------------|----------|---------------|------|
| 1.14.1-INT-035    | Integration | P0       | N8N sends callback with invalid embedding format → Verify rejection and error status | Critical: Data integrity |

**Scenario Details:**
```json
// Malformed embedding examples:
{
  "embeddings": null,  // Case 1: Null embeddings
  "embeddings": [],    // Case 2: Empty array
  "embeddings": [1, 2], // Case 3: Wrong dimensions (expected 1536)
  "embeddings": ["invalid"], // Case 4: Wrong type (string not float)
}
```

**Expected Behavior:**
- Validate embedding dimensions match model (e.g., 1536 for OpenAI)
- Validate all values are floats in range [-1, 1]
- Set status to 'failed' if validation fails
- Log detailed error for debugging

**Why P0:** Corrupt embeddings break RAG completely and are hard to detect.

### Category 3: Network Reliability & Timeouts

#### EC-008: Edge Function Timeout Before N8N Call

| ID                | Level       | Priority | Test Scenario | Risk |
|-------------------|-------------|----------|---------------|------|
| 1.14.1-INT-038    | Integration | P1       | Simulate edge function timeout (55s limit) before N8N webhook completes → Verify retry mechanism | Medium: Processing failure |

**Scenario Details:**
- **Supabase Limit:** Edge functions have 55-second timeout
- **Slow N8N:** If N8N response takes >55s, edge function dies
- **Expected:** Retry mechanism should re-invoke edge function
- **Validation:** Document eventually processes despite timeout

**Why P1:** Large documents might take longer to process.

#### EC-011: Signed URL Expires During N8N Processing

| ID                | Level       | Priority | Test Scenario | Risk |
|-------------------|-------------|----------|---------------|------|
| 1.14.1-INT-041    | Integration | P0       | Generate signed URL with short TTL (1min) → N8N processing takes 5min → Verify N8N gets 403 error but retries with new URL | Critical: Processing failure |

**Scenario Details:**
- **Problem:** Large documents take longer than signed URL TTL
- **Solution Options:**
  1. Increase signed URL TTL (e.g., 1 hour)
  2. Implement URL refresh mechanism in N8N
  3. Use permanent public URLs for documents (less secure)

**Expected Behavior:**
- Edge function should generate signed URL with TTL > max processing time
- Or N8N should request new URL if 403 received

**Why P0:** This would silently break processing for large documents.

## Key Insights

### Why So Many Integration Tests?

**Edge cases are inherently about interactions:**
- Race conditions require timing control (integration tests)
- Network failures require mocking external services (integration tests)
- Data corruption requires database transaction testing (integration tests)
- Security boundaries require multi-layer validation (integration tests)

Unit tests can't effectively validate these scenarios.

### Most Dangerous Edge Cases

1. **Silent Failures** (EC-005, EC-006, EC-011)
   - System appears to work but produces wrong results
   - Hard to detect without specific tests

2. **Race Conditions** (EC-001, EC-002, EC-003)
   - Async nature makes timing issues likely
   - Can cause data corruption or resource leaks

3. **Security Vulnerabilities** (EC-012, EC-014, EC-015, EC-022, EC-023)
   - Single vulnerability could compromise entire system
   - Must be tested before production

These 23 edge case scenarios complement the original 34 tests by focusing on **realistic production failures** that are unlikely but have high impact when they occur.
