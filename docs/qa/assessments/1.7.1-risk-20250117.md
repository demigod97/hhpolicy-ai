## Critical Risks Requiring Immediate Attention

### 1. SEC-001: Encryption Key Exposure

**Score: 9 (Critical)**
**Probability**: High (3) - Environment variables can be accidentally committed or exposed
**Impact**: High (3) - Complete compromise of all stored API keys

**Affected Components**:
- Environment variable management
- Encryption utilities (`src/utils/encryption.ts`)
- API key storage system

**Mitigation**:
- Store encryption key in secure vault (e.g., Supabase Vault)
- Never commit encryption keys to version control
- Implement key rotation mechanism
- Add pre-commit hooks to prevent accidental commits
- Use `.env.example` with placeholder values only

**Testing Focus**:
- Security audit of encryption key storage
- Penetration testing for key exposure vectors
- Code review for hardcoded secrets
- Environment variable validation tests

**Residual Risk**: Medium - Even with proper handling, environment misconfiguration possible
**Owner**: dev
**Timeline**: Before any production deployment

---

### 2. SEC-002: Insufficient Input Validation on API Keys

**Score: 9 (Critical)**
**Probability**: High (3) - Users may paste malformed or malicious strings
**Impact**: High (3) - Could lead to injection attacks or system compromise

**Affected Components**:
- `ApiKeyDialog.tsx` - Key input interface
- `src/services/apiKeyService.ts` - Key validation
- Database layer

**Mitigation**:
- Implement strict regex validation for API key formats
- Sanitize all inputs before encryption
- Validate key format per provider (OpenAI, Gemini, etc.)
- Add length limits and character whitelisting
- Implement rate limiting on key operations

**Testing Focus**:
- Fuzz testing with malformed inputs
- SQL injection testing
- XSS vulnerability testing
- Format validation unit tests

**Residual Risk**: Low - With comprehensive validation, attack surface minimized
**Owner**: dev
**Timeline**: Before deployment

---

### 3. SEC-003: Weak Encryption Implementation

**Score: 9 (Critical)**
**Probability**: Medium (2) - Crypto is hard; implementation errors common
**Impact**: High (3) - All stored keys compromised if encryption is weak

**Affected Components**:
- `src/utils/encryption.ts`
- AES-256 implementation
- Key derivation function

**Mitigation**:
- Use established crypto libraries (Web Crypto API or crypto-js)
- Implement proper IV (Initialization Vector) generation
- Use authenticated encryption (AES-GCM over AES-CBC)
- Add salt to key derivation
- External security audit of encryption code

**Testing Focus**:
- Cryptographic algorithm validation
- IV uniqueness verification
- Encryption/decryption round-trip tests
- Side-channel attack resistance

**Residual Risk**: Medium - Crypto implementation always carries risk
**Owner**: dev + external security audit
**Timeline**: Before production deployment

## High Risks

### 4. SEC-004: Insufficient Authorization Controls

**Score: 6 (High)**
**Probability**: Medium (2) - Authorization logic can be complex
**Impact**: High (3) - Users accessing keys they shouldn't

**Mitigation**:
- Implement RLS policies on `api_keys` table
- Verify System Owner can view all keys
- Ensure Company Operators only see their own keys
- Add comprehensive authorization tests
- Implement audit logging for all key access

**Testing Focus**:
- RLS policy validation
- Role-based access control testing
- Cross-tenant access prevention tests

---

### 5. DATA-001: Key Deletion Without Safeguards

**Score: 6 (High)**
**Probability**: High (3) - Users may accidentally delete active keys
**Impact**: Medium (2) - Service disruption if active key deleted

**Mitigation**:
- Implement soft delete pattern
- Require confirmation for deletion
- Prevent deletion of "default" keys without replacement
- Add warning if key has recent usage
- Implement key recovery mechanism (30-day retention)

**Testing Focus**:
- Deletion workflow validation
- Confirmation dialog testing
- Soft delete verification
- Recovery mechanism testing

---

### 6. PERF-001: Encryption Performance on High Volume

**Score: 6 (High)**
**Probability**: Medium (2) - Encryption operations are CPU intensive
**Impact**: High (3) - System slowdown under load

**Mitigation**:
- Cache decrypted keys in memory (with TTL)
- Implement connection pooling
- Use asynchronous encryption operations
- Load test with realistic key volumes
- Monitor encryption operation latency

**Testing Focus**:
- Load testing with 1000+ concurrent key operations
- Encryption latency benchmarks
- Memory usage under load
- Cache effectiveness metrics

---

### 7. OPS-001: Missing Key Rotation Mechanism

**Score: 6 (High)**
**Probability**: High (3) - Manual key rotation is error-prone
**Impact**: Medium (2) - Stale keys pose security risk

**Mitigation**:
- Add key expiration dates
- Implement automated rotation reminders
- Create key rotation workflow
- Track key age in UI
- Notification system for expiring keys

**Testing Focus**:
- Expiration logic validation
- Rotation workflow testing
- Notification delivery verification
