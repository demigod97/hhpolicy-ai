## Technical Approach

### Phase 1: Audit & Preparation (2 hours)

```sql
-- Audit script to find all references
SELECT
    tc.table_name,
    tc.constraint_name,
    tc.constraint_type,
    kcu.column_name,
    ccu.table_name AS foreign_table_name,
    ccu.column_name AS foreign_column_name
FROM information_schema.table_constraints AS tc
JOIN information_schema.key_column_usage AS kcu
  ON tc.constraint_name = kcu.constraint_name
JOIN information_schema.constraint_column_usage AS ccu
  ON ccu.constraint_name = tc.constraint_name
WHERE tc.table_name = 'notebooks'
   OR ccu.table_name = 'notebooks';

-- Find all functions referencing notebooks
SELECT routine_name, routine_definition
FROM information_schema.routines
WHERE routine_definition LIKE '%notebooks%';

-- Find all triggers
SELECT trigger_name, event_manipulation, event_object_table
FROM information_schema.triggers
WHERE event_object_table = 'notebooks';
```

### Phase 2: Create Migration Script (1 hour)

**File**: `supabase/migrations/20251019000000_rename_notebooks_to_chat_sessions.sql`

```sql
-- Migration: Rename notebooks table to chat_sessions
-- Date: 2025-10-19
-- Author: PolicyAi Team

BEGIN;

-- Step 1: Rename the main table
ALTER TABLE notebooks RENAME TO chat_sessions;

-- Step 2: Rename foreign key columns in dependent tables
ALTER TABLE notes
  RENAME COLUMN notebook_id TO chat_session_id;

ALTER TABLE n8n_chat_histories
  RENAME COLUMN notebook_id TO chat_session_id;

-- Step 3: Update foreign key constraint names (if needed)
-- PostgreSQL automatically updates constraint names, but we can be explicit

-- Step 4: Rename indexes to reflect new table name
ALTER INDEX IF EXISTS notebooks_pkey
  RENAME TO chat_sessions_pkey;

ALTER INDEX IF EXISTS notebooks_user_id_idx
  RENAME TO chat_sessions_user_id_idx;

ALTER INDEX IF EXISTS notebooks_created_at_idx
  RENAME TO chat_sessions_created_at_idx;

-- Step 5: Update RLS policies
-- Drop old policies (they're tied to old table name)
DROP POLICY IF EXISTS "Users can view their own notebooks" ON chat_sessions;
DROP POLICY IF EXISTS "Users can create their own notebooks" ON chat_sessions;
DROP POLICY IF EXISTS "Users can update their own notebooks" ON chat_sessions;
DROP POLICY IF EXISTS "Users can delete their own notebooks" ON chat_sessions;

-- Recreate policies with new names
CREATE POLICY "Users can view their own chat sessions"
  ON chat_sessions FOR SELECT
  USING (auth.uid() = user_id);

CREATE POLICY "Users can create their own chat sessions"
  ON chat_sessions FOR INSERT
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update their own chat sessions"
  ON chat_sessions FOR UPDATE
  USING (auth.uid() = user_id)
  WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can delete their own chat sessions"
  ON chat_sessions FOR DELETE
  USING (auth.uid() = user_id);

-- Step 6: Update database functions that reference notebooks
-- Example: If there's a function like get_user_notebooks()
-- This will vary based on actual functions in the database

-- Step 7: Verify record count
DO $$
DECLARE
  record_count INTEGER;
BEGIN
  SELECT COUNT(*) INTO record_count FROM chat_sessions;
  RAISE NOTICE 'Total chat_sessions records: %', record_count;
END $$;

COMMIT;
```

## Notes

### Why This Change?

The term "notebook" is legacy from InsightsLM where users created "notebooks" for different topics. In PolicyAi, users create **chat sessions** to ask questions about policies. The database should reflect this semantic clarity.

### Alternative Approaches Considered

1. **Keep "notebooks" name**: Rejected - creates confusion for new developers
2. **Create new table, migrate data**: Rejected - unnecessary complexity, same result
3. **Use database views**: Rejected - adds layer of indirection, doesn't solve core issue

## Implementation Decision - January 19, 2025

After analysis of the existing codebase and database schema, the team has decided to **DEFER** this migration for the following reasons:

### Why We're Keeping `notebooks` Table Name

1. **Backward Compatibility with N8N Workflows**
   - Existing n8n chat workflows reference the `notebooks` table
   - Migration would require updating all n8n workflows
   - Risk of breaking existing chat functionality

2. **Minimal Impact on User Experience**
   - Table naming is internal implementation detail
   - Users never see the table name `notebooks`
   - UI already uses correct terminology ("Policy Documents", "Chat Sessions")

3. **Code Already Uses Correct Abstractions**
   - Frontend components use semantic names (DocumentGrid, ChatSession, etc.)
   - Hooks and services abstract the database table names
   - TypeScript interfaces use correct domain language

4. **Technical Debt is Manageable**
   - Database migration exists: `20251019000000_fix_chat_sessions_notebook_reference.sql`
   - This migration ensures `chat_sessions` table correctly references `notebooks`
   - Comment added to explain: "References notebooks table for backward compatibility with existing n8n chat system"

### What Was Actually Implemented

Instead of renaming `notebooks` → `chat_sessions`, we:
- Created a separate `chat_sessions` table for native chat functionality
- Added foreign key from `chat_sessions.notebook_id` → `notebooks.id`
- Documented the relationship clearly in migration comments
- Maintained backward compatibility with existing systems

### Stories 1.14.2 and 1.14.3 Proceed Without This

Stories 1.14.2 (Dashboard PDF Document Grid) and 1.14.3 (React-PDF Viewer Integration) have been successfully implemented WITHOUT requiring the `notebooks` → `chat_sessions` rename.

The Dashboard now displays PDF policy documents from the `sources` table, not from `notebooks`, so this migration is no longer a blocker.

### Future Consideration

This migration could be reconsidered in the future when:
- N8N workflows are migrated to new architecture
- All chat functionality is migrated to `chat_sessions` table
- Risk/benefit analysis shows clear value

For now, the table name remains `notebooks` with clear documentation explaining its purpose.
