# Story 1.17.1: Dashboard Interactive Document Table

**Epic**: 1.17 - Dashboard Enhancement & UX Improvements
**Priority**: 🔴 **HIGH**
**Status**: 📋 **PLANNED**
**Effort**: 3-4 days
**Created**: October 28, 2025

## User Story

**As a** PolicyAi user (Administrator, Executive, Board Member, System Owner, or Company Operator)
**I want to** view policy documents in an interactive table with advanced features instead of cards
**So that** I can efficiently browse, search, filter, sort, and perform bulk actions on policy documents with better data density and management capabilities.

---

## Context & Motivation

### Current State (Card-Based Grid)
The Dashboard currently uses a card-based grid view (`DocumentGrid.tsx`) that displays documents as individual cards:
- Limited information density (requires scrolling for many documents)
- No bulk selection capabilities
- Limited sorting/filtering options
- Inefficient for managing large document sets
- No advanced table features (column sorting, pagination, bulk actions)

### Desired State (Interactive Table)
Transform the document view into a feature-rich interactive table with:
- High information density (view more documents at once)
- Bulk selection with checkboxes
- Advanced filtering and sorting
- Column-based organization
- Inline actions and management features
- shadcn/ui Table components for consistency

### Business Value
1. **Efficiency**: Manage multiple documents simultaneously with bulk operations
2. **Scalability**: Better UX for organizations with 50+ policy documents
3. **Usability**: Industry-standard table interface familiar to enterprise users
4. **Power User Features**: Advanced sorting, filtering, and data manipulation

### Data Model & Type Definitions

```typescript
// Enhanced Document type for table
interface DocumentTableRow {
  id: string;
  title: string;
  policyType: string | null;
  policyDate: string | null;
  policyDateParsed: Date | null;
  isOutdated: boolean;
  targetRole: 'administrator' | 'executive' | 'board';
  processingStatus: 'pending' | 'processing' | 'completed' | 'failed';
  uploadedAt: string;
  uploadedBy: string;
  fileSize: number;
  pageCount: number | null;
  pdfFilePath: string;
  pdfStorageBucket: string;
}

// Filter state
interface DocumentFilters {
  search: string;
  roles: string[];
  policyTypes: string[];
  dateRange: { start: Date | null; end: Date | null };
  statuses: string[];
  showOutdatedOnly: boolean;
}

// Sort state
interface DocumentSort {
  column: keyof DocumentTableRow;
  direction: 'asc' | 'desc';
}

// Table state
interface DocumentTableState {
  filters: DocumentFilters;
  sort: DocumentSort;
  selectedRows: Set<string>;
  page: number;
  pageSize: number;
}
```

### Database Queries

**Enhanced useDocuments Hook**:
```typescript
// src/hooks/useDocuments.tsx
export const useDocuments = (filters?: DocumentFilters, sort?: DocumentSort) => {
  const { userRole } = useUserRole();

  return useQuery({
    queryKey: ['documents', userRole, filters, sort],
    queryFn: async () => {
      let query = supabase
        .from('sources')
        .select(`
          id,
          title,
          type,
          target_role,
          processing_status,
          created_at,
          uploaded_by_user_id,
          file_size,
          pdf_file_path,
          pdf_storage_bucket,
          metadata,
          policyType,
          policyDate,
          policyName
        `)
        .eq('type', 'pdf');

      // Apply RLS filtering (handled by Supabase)
      // Apply frontend filters
      if (filters?.search) {
        query = query.ilike('title', `%${filters.search}%`);
      }
      if (filters?.roles?.length) {
        query = query.in('target_role', filters.roles);
      }
      if (filters?.statuses?.length) {
        query = query.in('processing_status', filters.statuses);
      }

      // Apply sorting
      if (sort) {
        query = query.order(sort.column, { ascending: sort.direction === 'asc' });
      } else {
        query = query.order('created_at', { ascending: false });
      }

      const { data, error } = await query;
      if (error) throw error;

      // Transform to table row format
      return data.map(transformToTableRow);
    },
    enabled: !!userRole,
  });
};
```

## Notes

### Design Decisions

1. **Why Table Over Cards?**
   - Higher information density for large document sets
   - Industry standard for enterprise document management
   - Better support for bulk operations
   - Easier scanning and comparison

2. **Column Selection Rationale**
   - Document Name: Primary identifier, clickable
   - Level (PolicyType): Critical for categorization
   - Policy Date: Compliance requirement with outdated indicators
   - Target Role: Access control visibility
   - Status: Real-time processing feedback
   - Uploaded: Recency indicator
   - Actions: Quick access to operations

3. **Pagination vs Infinite Scroll**
   - Pagination chosen for predictability and performance
   - Easier to reference specific documents ("Page 3, row 5")
   - Better for accessibility
   - Can add virtualized scrolling later if needed

4. **Bulk Actions Priority**
   - Delete: Most common operation for cleanup
   - Change Role: Essential for access control management
   - Export: Useful for reporting (future enhancement)

### Technical Considerations

- **Performance**: Monitor query performance with complex filters
- **RLS**: Ensure Supabase RLS policies handle table queries efficiently
- **Type Safety**: Update Supabase generated types for new fields
- **State Management**: Consider Zustand if table state becomes complex

### User Feedback Integration

- Gather feedback during beta release
- Monitor analytics for feature usage
- Iterate on filter/sort options based on user needs
- Consider A/B testing table vs card view (if toggle implemented)
