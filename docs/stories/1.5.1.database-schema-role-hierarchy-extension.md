# Story 1.5.1: Database Schema - Add Company Operator & System Owner Roles

## Status: Done

## Story

As a System Owner,
I want the database to support 5 user roles instead of 3,
so that we can assign Company Operator and System Owner roles to users for operational management.

## Context Source

- Source Document: Sprint Change Proposal - Epic 1.5
- Enhancement Type: Database schema extension
- Existing System Impact: Extends current 3-role system to 5-role hierarchy

## Acceptance Criteria

1. Database migration script extends the user_roles constraint from `('board', 'administrator', 'executive')` to include `('company_operator', 'system_owner')`.
2. Role assignment functionality allows assignment of all 5 roles.
3. Existing users with 3 original roles remain unaffected by the migration.
4. RLS policies are updated to recognize and enforce permissions for the 2 new roles.
5. New indexes are created for company_operator and system_owner roles for performance.
6. All existing functionality continues to work with the extended role system.

## Dev Technical Guidance

### Database Schema Changes

**Migration File:** `20251016145126_extend_user_roles_for_operators.sql`

**Key Changes:**

- Extend user_roles constraint to include new roles
- Update policy_documents and sources constraints
- Add 8 new RLS policies for new roles
- Create indexes for company_operator and system_owner
- Maintain backward compatibility

**New Role Hierarchy:**

1. **System Owner** - Full system control, user limit management
2. **Company Operator** - User role management, API key configuration
3. **Board** - Full document access, read-only
4. **Administrator** - Policy management, document upload
5. **Executive** - Standard access, document viewing

### RLS Policy Updates

**New Policies Required:**

- Company Operator access to user_roles table
- System Owner access to all tables
- Role-based document access for new roles
- API key management permissions
- Token usage monitoring permissions
