# Story 1.5.2: Update RLS Policies for 5-Tier System

## Status: Approved

## Story

As a System,
I want all database tables to enforce row-level security for the 5-role hierarchy,
so that users can only access data appropriate to their role level.

## Context Source

- Source Document: Sprint Change Proposal - Epic 1.5
- Enhancement Type: Security policy extension
- Existing System Impact: Extends current RLS policies to support new roles

## Acceptance Criteria

1. RLS policies on `policy_documents` table allow Company Operators to see all documents for upload/management.
2. RLS policies on `user_roles` table allow Company Operators to view and modify roles (excluding System Owner role).
3. RLS policies on new `api_keys`, `token_usage`, `user_limits` tables enforce role-based visibility.
4. System Owners have full access to all tables via RLS policies.
5. Board Members retain read-only access to policy documents.
6. Comprehensive test suite validates RLS policy enforcement for all 5 roles.

## Dev Technical Guidance

### RLS Policy Hierarchy

**System Owner (Highest Level):**
- Full access to all tables
- Can manage user limits
- Can assign any role including other System Owners
- Can access all system configuration

**Company Operator:**
- Can manage user roles (except System Owner)
- Can configure API keys
- Can view token usage across company
- Can upload documents for any role
- Cannot assign System Owner role

**Board Member:**
- Read-only access to all policy documents
- Can view company-wide document library
- Cannot modify user roles or system settings

**Administrator:**
- Can upload and manage Administrator documents
- Can assign documents to Executive role
- Cannot access Company Operator or System Owner functions

**Executive:**
- Can view Executive and Board documents
- Cannot upload documents
- Cannot access administrative functions

### New Tables Requiring RLS Policies

**api_keys table:**
- Company Operators: Can manage keys they created
- System Owners: Can manage all keys
- Others: No access

**token_usage table:**
- Company Operators: Can view company-wide usage
- System Owners: Can view all usage and manage limits
- Users: Can view their own usage only

**user_limits table:**
- System Owners: Full access to manage limits
- Company Operators: Read-only access to view limits
- Others: No access

**chat_sessions table:**
- Users: Can access their own sessions
- Company Operators: Can view company sessions for monitoring
- System Owners: Can view all sessions

### Testing Requirements

**Security Tests:**
- Verify role-based access enforcement
- Test unauthorized access attempts
- Validate data isolation between roles
- Test edge cases and boundary conditions

**Performance Tests:**
- RLS policy performance impact
- Query execution with role filtering
- Index usage with RLS policies
