# Story 1.7.1: API Key Management System

## Status: Approved

## Story

As a Company Operator,
I want to securely store and manage API keys for AI providers,
so that our company can use our own API keys instead of relying on system defaults.

## Context Source

- Source Document: Sprint Change Proposal - Epic 1.7
- Enhancement Type: API key management system
- Existing System Impact: New secure storage and management for API keys

## Acceptance Criteria

1. API Key Manager UI allows adding keys for OpenAI, Gemini, Mistral, and Anthropic providers.
2. Keys are encrypted using AES-256 before storage in `api_keys` table.
3. Only key preview (last 4 characters) is displayed in the UI, never the full key.
4. "Test Key" functionality validates key with provider API before saving.
5. Users can set one key per provider as "default" for automatic selection.
6. Key usage count and last used timestamp are tracked automatically.
7. Company Operators can activate/deactivate keys without deletion.
8. System Owners can view all company API keys; Company Operators can only manage keys they created.
9. Encryption key is stored in environment variables, not in database.

## Dev Technical Guidance

### API Key Management Architecture

**Database Schema:**
- `api_keys` table with encrypted storage
- AES-256 encryption for key storage
- SHA-256 hash for verification
- Usage tracking and timestamps

**Security Implementation:**
- Keys encrypted at rest using AES-256
- Encryption key in environment variables
- Never expose full keys in frontend
- Audit logging for key operations

**Provider Support:**
- OpenAI (GPT models)
- Gemini (Google AI)
- Mistral (Mistral AI)
- Anthropic (Claude models)

**Key Operations:**

### Critical Security Requirements

Before implementation begins, the following MUST be addressed:

1. **Encryption Key Storage**
   - ❌ Do NOT use environment variables for encryption keys
   - ✅ MUST use Supabase Vault or equivalent secure key management
   - ✅ MUST implement key rotation mechanism
   - ✅ MUST add pre-commit hooks to prevent key exposure

2. **Input Validation**
   - ✅ MUST implement provider-specific regex validation
   - ✅ MUST sanitize all inputs before encryption
   - ✅ MUST add length limits and character whitelisting
   - ✅ MUST implement rate limiting on key operations

3. **Encryption Implementation**
   - ✅ MUST use established crypto libraries (Web Crypto API or crypto-js)
   - ✅ MUST use authenticated encryption (AES-GCM preferred over AES-CBC)
   - ✅ MUST implement proper IV generation (unique per encryption)
   - ✅ MUST add salt to key derivation
   - ✅ MUST schedule external security audit before production

4. **Authorization & RLS**
   - ✅ MUST implement comprehensive RLS policies on api_keys table
   - ✅ MUST test System Owner can view all company keys
   - ✅ MUST test Company Operators only see their own keys
   - ✅ MUST prevent cross-tenant access

5. **Data Protection**
   - ✅ MUST implement soft delete pattern with 30-day retention
   - ✅ MUST add confirmation dialogs for deletion
   - ✅ MUST prevent deletion of default keys without replacement
   - ✅ MUST implement key recovery mechanism

6. **Performance Validation**
   - ✅ MUST conduct load testing with 1000+ concurrent operations
   - ✅ MUST implement key caching strategy with TTL
   - ✅ MUST monitor encryption operation latency
   - ✅ MUST validate memory usage under load
